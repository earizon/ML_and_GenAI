# AI Agents

  Agentic systems promise to extend the possibility of computer programs 
  beyond the mere execution of pre-determined workflows conceived to 
  solve narrow tasks. <br/>
   In fact, most real-life problems do not fit in pre-determined workflows.
  Example: think of the infinite possibilities of GNU Text utils 
     <https://www.gnu.org/software/coreutils/manual/coreutils.html>
    vs the limited set of well-known or enterprise deployed scripts making
  use of them for particular use cases or scenarios.

  Cite by Roucher, Noyan, and Wolf (Hugging Face Smolagents engineers):

> If a deterministic workflow fits all queries, by all means just code 
> everything! This will give you a 100% reliable system with no risk of error 
> introduced by letting unpredictable LLMs meddle in your workflow. For the sake 
> of simplicity and robustness, it's advised to regularize towards not using any 
> agentic behavior.
  [[qa.101.dont's]]

* An AI Agent can be described as an intelligent system designed to interact 
  with an environment.  Examples include those in a self-driving car, a chatbot, 
  intelligent characters in moderm video games, More formally an Agent is an
  entity capable of four key actions:
  - Perceiving its environment: robot identifying "space time", or software
    aware of files in a hard-disk.
  - Processing information: doing request to an API, ...
  - **Taking action to achieve specific goals**: Setting up heating room,
    searching for related content, ...
  - **Adapting its behavior based on experience**: Re-try when finding an error, 
    ...
* AI agents can be categorized into different types, such as:
  - **Reactive Agents**: These agents respond to specific stimuli in their 
    environment without maintaining an internal state or memory.
  - **Deliberative Agents**: These agents maintain an internal model of the 
    world and can plan and reason about their actions.
  - **Learning Agents**: These agents can improve their performance over time 
    by learning from data and experiences.
  - ...

* An agent-based system can also be split or categorized as:
  - multi-step single agent.
  - multi-agent, with each agent "prompt set of knowledge" developed in
    isolation by different groups.


* Agentic AI refers (as of 2026) to AI Agents using LLM architectures.

* Year 2022 adn 2023 marked a turning point in AI agents with the introduction 
  of techniques like Chain of Thought ("think step by step" system prompt suffix),
  Reflection, and ReAct (Reasoning-Acting) unlocking powerful new capabilities
  that are foundational to today's advanced AI products. 
  * "Reasoning" in the context of LLMs, is refered to the way the LLM breaks
    down a task when adding the "think step by step". **It is the most important
    part off an agentic AI system**.
  * "ReAct refers to the ability of the AI Agent to deconstruct a problem, take
    sequential action, and utilize external tools to achieve a goal.

* Core Components of an Agentic AI System:
  - Planning: Creating a step-by-step approach to solve a problem.
  - Tools   : Identifying and using the right tools.
  - Action  : Executing steps by making programmatic calls and retrieving information.
  - Memory  : Keeping track of learned information to make an informed final decision.
    * NOTE: LLMs are stateless, i.e, they don't remember past prompts or prompt answers
      unless that information is explicitly included.  To behave as "smart" agents,
      they need a simulated form of memory commonly referred to as "short-term memory".
    * Memory Is a Simulation: What looks like memory is actually constructed by
      including previous interactions in the current prompt creating the illusion of
      continuity. The agent doesn’t remember; it’s simply reminded. [[doc_has.keypoint]]<br/>
      Memorry simulation APIs ussualy contain methods like "pop()" to remove last messages,
      "reset()" to clean memory, "shift()" to forget old messages, ...
    * There are 2 types of memory:
      - Short-term (Ephemeral) memory (In-application): Exists only during the chat session.
      - Long -term (Durable  ) memory : Persisted in databases, vector stores, ...  and 
        available across sessions (days, months, ...). Use to store essential facts,
        (long term "true claims", user preferences, ...).<br/>
        - Long-term memory typically falls into three categories:
          - Semantic Memory: For factual information (example: user profiles - preferences -.
            often stored in structured formats.
          - Episodic Memory: Sotring past interactions and events. Useful to create few-shot
            learning by recalling sucessful examples from past interaction.
          - Procedural Memory: Encoding and storing behavioral adaptations. [[PM.TODO]]
        - Challenges and Best Practices: [[qa.best_patterns]].
          - balances memory data vs relevance of data.
          - Set expiration time whem possible.
          - Create memory "namespaces" (by taxonomy groups, projects, ...)
          - Consider privacy issues 
        - NOTE a long-term memory that does not expires can end up being part of the
          "absolute true" in the RAG.


    * NOTE: State (of an State Machine) can also be seen as an specialized structured 
      memory. This applies to agentic flows designed around state machines.
      This is a state controlled by the code invoking the LLM (vs a simulated prompt
      with reminded history used by LLM).

* Agentic AI is more "sexy" than previous Enterprise Automation tools (scripted workflows,
  Robotic Process Automation (RPA), AI workflows, ...) because it its the first one to
  **independently create planning a sequence of steps without human intervention**.

* Reasoning LLMs main difference is that they are trained to expose their "internal
  monologue" (or thought) to the user. It has been trained to:
  1. Create Initial Ceconnaissance.
  2. Forming an Hypothesis.
  3. Identifying a Problem.
  4. Attempting to Correct.
  5. Course Correction (Pivoting). "leasson learned: I prematurelñy chose target B before
     target A being ready".
  6. Revise Plan.
  7. Create Final Plan.

* As a (human) develop the focus in constructing new agest is to map real-world business
  tasks into a series of agent-executable steps, complete with decision points, tool 
  integrations, and context handling.

* Two "extremes" AI Agents exists:
  * "full agency" agents, with agents operating on their own derived plans.
  * "on-rails" workflows, where humans remain in the loop. (most common as of 2026)


### Agentif Workflow building blocks

All the following 5 building blocks must be in place for an agent to work properly.

1. Breaking large goals into "thin" tasks.
2. Proper task ordering.
3. Decision Points: Map out moments when the agent must evaluate information 
   and choose a path (business event that must trigger a path selection).<br/>
     Decision nodes allow the agent to handle the unexpected and "short-circuit"
   path to completion.
4. Identification and integration of external tools. (REST APIs, File System, ...).
5. Context memory management keeping track of outputs from "executed" tasks.

### Agentic AI Evaluation-driven Requirements Documentation


* An Agentic AI requirements document is formed by a testable hypothesis: 
  **Success (business goal) can be measured and failure can be debugged**.
* A good requirements document creates alignment across teams. A great one 
  bakes in its own evaluation plan. You are not just scoping what to build; 

  ```
  | Example 1: 
  | 
  | Given this context, this set of tools and this target goal, the agent should be able to achieve XXX." 
  ```

  ```
  | Example 2: 
  | 
  | # Objective:
  | 
  | Automate 90% of manual onboarding tasks, aiming for a 30% reduction in new employee ramp-up time.
  | 
  | # Workflow Summary
  | 
  | Create internal account.
  | Complete HHRR contact data.
  | Evaluate employee clearance.
  | Create 3rd party account (mail, ...)
  | Provide resources to employee (laptop, mobile).
  | Send welcome pack.
  | confirm task completion.
  | 
  | # Capabilities:
  | 
  | Must maintain state across sessions.
  | Must escalate to HR if a blocker persists for more than 2 days.
  | 
  | # Tool Integrations
  | 
  | Requires access to the HRIS API, IT provisioning tool, and LMS webhook.
  | 
  | # Success Metrics
  | 
  | Average time-to-full-access < 2 days.
  | agent-led task completion rate > 85%.
  | 
  |
  | # Guardrails/Constraints.
  |
  | 1. Agent must not access Personal Information (PII) beyond specified onboarding fields.
  | 2. Agent must not execute DDBB queries. Just put then in a query for human validation.
  | 3. Agent must fall back to HR after two failed attempts at a task.
  ```

* No matter the method, four core components are required:
  - Inputs        : User queries, tool options, memory, and instructions.
  - Outputs       : Agent responses and tool call sequences.
  - Reference Data: Ground truth answers or expected actions.
  - Evaluators    : Rules, heuristics, or models (like an LLM) to assess success.

### Basic AI Agent QA

* AI Agents and LLMs Evaluation Sets ("Evals"):<br/>
  * Structured tests designed to estimate how well the system is performing.
  * Input comes from domain experts and business professionals who 
    understand the nuances of the tasks the agent is meant to 
    perform. (comming back to expert systems, but just for testing
    performance -vs full training-)  [[qa.evaluation_sets]]
* Guardrails: mechanisms designed to steer the agent away from potentially
  harmful paths by verifying the model's response **before being sent to users**.
  Examples:
  * Pass the agent's response through a content moderation filter.
  * Check for and remove any personal information before it is shared.
  * Automatically escalate an issue to a human representative if
    a customer expresses a high-risk concern.
* Specific Safety Evals: testing system's resilience against prompt attacks
  (malicious querie3s).

### System 1 vs System 2 thinking:

Based on the physcologist Daniel Kahneman's framework:

* System 1 Thinking: A mode of thinking that is fast, automatic, intuitive, and reactive. 
  ("simple" or "standard" LLMs often exhibit this type of response).
* System 2 Thinking: A mode of thinking, that is slow, deliberate, conscious, and
  controlled. (LLM Reasoning models)

  ```
  | $ pip install networkx                            # Presetup    
  | $ cat example_DAG.py --------------------------------------------------------------------------
  | import networkx as nx
  |  
  | dag = nx.DiGraph()                              # 1. Create a directed graph
  |  
  | dag.add_nodes_from([                            # 2. Add tasks
  |     "set_table", "cook_food",
  |     "prepare_drinks", "serve_dinner" ])
  |  
  | dag.add_edge("set_table", "serve_dinner")       # 3.1 Add dependencies
  | dag.add_edge("cook_food", "serve_dinner")
  | dag.add_edge("prepare_drinks", "serve_dinner")
  | if not nx.is_directed_acyclic_graph(dag):       # 3.2 Validate DAG (No Cycles)
  |  
  |     raise ValueError("Graph contains a cycle")
  | execution_order = list(nx.topological_sort(dag))# 4. Get Execution Order (Topological Sort)
  |  
  | def execute_dag(graph, task_handlers):          # 5. Simple DAG Orchestrator
  |     completed = set()
  |  
  |     for node in nx.topological_sort(graph):
  |         dependencies = set(                     # Check: Ensure all dependencies are complete
  |             graph.predecessors(node))
  |         if not dependencies.issubset(completed):
  |             raise RuntimeError(
  |               f"Dependencies not met for {node}")
  |         task_handlers[node]()
  |         completed.add(node)
  |  
  | def set_table():                                # 6. Task Implementations
  |     print("Table is set")
  | def cook_food():
  |     print("Food is cooked")
  | def prepare_drinks():
  |     print("Drinks are ready")
  | def serve_dinner():
  |     print("Dinner is served")
  |  
  | handlers =
  | execute_dag(dag, {                               # 7. Run the Workflow
  |     "set_table"     : set_table,                 # > Table is set
  |     "cook_food"     : cook_food,                 # > Food is cooked
  |     "prepare_drinks": prepare_drinks,            # > Drinks are ready
  |     "serve_dinner"  : serve_dinner               # > Dinner is served
  | })
  ```

## RACI Model or RACI (multi-agent) Pattern

The RACI model defines four key roles in project management or multi-agent architectures:
- Responsible: person, team or AI Agent who actually does the work to complete the task.
- Accountable: Individual (or Agent "Judge" who is ultimately answerable for the correct and thorough completion of the task.
               This person/agent **delegates** the work and ensures it is done.
- Consulted  : Those whose opinions are sought.<br/>
               Typically subject matter experts. This role involves two-way communication.
- Informed   : Individuals who need to be kept updated on progress or decisions but do not need to be consulted directly. 
               (or Agents receiving events of interest)


This matrix helps clarify roles and responsibilities, ensuring everyone understands their part in the project.
