# Spec Driven development (SDD)


Problem: Vibe coding gives quick results but it doesn't scale to a large ongoing project.

Solution?: Spec-Driven Development tries to bring back engineering to "prompt-guided" software development.


Core idea: Maintain an evolving specification as a set of "markdown" files (sort of "big context")
used as Agent context input and handling permanent (permanent == sort of "system prompt" like) technical artifact.

- The Specification in SDD writes down the "what and why", decoupled from the Implementation, ("the how")
- It defines the set of contracts between the humans, and humans and agents.

  ```
  | INTENTIONS    ···············>  SPECIFICATION  ················> IMPLEMENTATION
  | AND INTERESTS └──────┬───────┘    ^            └───────┬───────┘
  |                      ·            ·                    ·    
  |              humans with the      ·              Code assitants with
  |              help of Agent/s.     ·              human supervision
  |              (engineers,...)      ·
  |                                   ·
  |                                   specs/mission.md
  |                                   specs/tech-stack.md
  |                                   specs/roadmap.md
  |                                   specs/spec_xxx_.../plan.md
  |                                   specs/spec_xxx_.../requirements.md 
  |                                   specs/spec_xxx_.../validation.md
  |                                   skills/...
  |                                   ... (backlog.md, ...)
  ```

- SSD main Benefits:
  1. Get back in control of large code changes with small changes to the spec.
  2. Specs eliminate the context decay problem (typical in multi-turn agent sessions).<br/>
     - Specs ("core context") persist between sessions and even agents.
       Can we see the specs as a "system-book" vs a "system-prompt"?
  3. Improve intent fidelity. Before the agent starts generating code, the Specs
     force humans to:
     - to define the problem.
     - success criteria.
     - constraints.
     - user flows.
     - ...

## Workflow overview


up front for the project, then for each feature, getting better as you go.

SDD Workflow: humans help agent with the best quality context ("system book" vs "system prompt"):
1. decisions about the project.
2. details about each feature.

TIP: treat your agent as a highly capable pair programmer to balance the right level of detail.
- Provide "Lots" of context about the Goals, mission, target audience and constraints.
  and less about low-level code decisions the agent can figure out on its own.
