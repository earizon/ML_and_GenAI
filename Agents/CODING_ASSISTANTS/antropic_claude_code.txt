```
  .claude/hooks/pre-commit.sh               <·· STEP 2: ~10 minutes
   if git diff --cached --name-only | \         Example: block commits that contain .env files.
      grep -qE '\. env$'; 
   then
       echo "BLOCKED: .env file detected"  exit 1 
   fi 


   claude mcp add mistral \                 <·· STEP 3: Connect MCP server at will (~10-15min)
      -- npx -y mistral-mcp@latest              
                                                
   /mcp                                         <·· see what's connected and how tokens it does consume.                
                                            

   .claude/agents/test-runner/AGENT.md      <·· STEP 4: A Subagent runs in its own CLEAN CONTEXT and returns
                                                only a summary avoiding to polute the main context.
                                                - Give your sub-agent one well defined job: 
                                                  "run test suite, parse failures, return short report"
   ./CLAUDE.md.                             <·· ~20-60 minutes. This is the project memory Claude reads
     ^                                          on **every session** (every "chat").
     │                                          Do not overfill it, update when needed.
     └─ A good CLAUDE.md has 4 sections:
        1. How to run the project (commands, ports, env)
        2. Code style rules adopted by our team.
        3. What NOT to touch / NOT to do. (legacy files, third-party code)
        4. Current priorities (what you're actively working on)
   - Claude works 3x better with a good CLAUDE.md. [[agents.claude_code.101]]
```

## Claude Batch mode how-to

```
| [[{]] 
|   CLAUDE_OPTS=""
|   CLAUDE_OPTS="${CLAUDE_OPTS} --permission-mode acceptEdits"               # ¹
|   CLAUDE_OPTS="${CLAUDE_OPTS} --allowedTools "Bash(git *) Edit Write Read" # <·· Allow some tools (Agentic mode)
|   CLAUDE_OPTS="${CLAUDE_OPTS} --disallowedTools "Bash(rm *)"               # <·· disallow (Guardrail)
|   CLAUDE_OPTS="${CLAUDE_OPTS} --model claude-sonnet-4-6"                   # 
|   CLAUDE_OPTS="${CLAUDE_OPTS} --max-turns 3 "                              # 
| # CLAUDE_OPTS="${CLAUDE_OPTS} --output-format json"                        # := json | stream-json for Machine-readable output
| # CLAUDE_OPTS="${CLAUDE_OPTS} --session-id <uuid> / --resume "             # Resumable runs            
| # CLAUDE_OPTS="${CLAUDE_OPTS} --add-dir /path "                            # Grant access to extra dirs
| # CLAUDE_OPTS="${CLAUDE_OPTS} --mcp-config servers.json "                  # Load MCP servers          
| # CLAUDE_OPTS="${CLAUDE_OPTS} --max-budget-usd 3.00"                       # Cost cap                  
| # CLAUDE_OPTS="${CLAUDE_OPTS} " # 
| 
| PROMPT="Generate TS types for our API and write to ./output/types.ts"
| echo ${PROMPT} |   claude --print ${CLAUDE_OPTS} > result.json             # <·· -p/--print => print and exit (batch mode)
| echo $?  # 0 => success, 1 => error (API / tool / permission), 2 => pre-tool-use hook blocked.
|
| ¹ Permission mode values:
|   - default           (unusable in batch).
|   - acceptEdits       (auto-approve edits in CWD — recommended)
|   - bypassPermissions (skip all — dangerous).
|   - plan              (read-only)
|   - dontAsk           (deny unless allow-listed).
| 
| Alternatively edit .claude/settings.json:
|
|   + "permissions": {
|   +   "allow": [
|   +     "Bash(git *)",
|   +     "Bash(npm run *)",
|   +     "Edit(/src/**)",
|   +     "Write(/output/**)",
|   +     "Read"
|   +   ],
|   +   "deny": ["Bash(rm *)", "Bash(curl *)"]
|   + },
|
| [[security.101]]
| 
|   - Avoid --dangerously-skip-permissions outside sandboxed/disposable environments.
|   - Prefer acceptEdits + an explicit permissions.allow list over bypassPermissions.
|   - Settings precedence: managed → CLI args → project .claude/settings.json → user ~/.claude/settings.json → defaults.
| 
| REF:
| - <https://code.claude.com/docs/en/permissions.md>
| - <https://code.claude.com/docs/en/settings.md>
| - claude --help
| [[}]] 
```

##  code-review-graph: Handle large codebase


(Summary from post by Wassim RekikWassim Rekik on LinkedIn)

* <https://github.com/tirth8205/code-review-graph>

* `code-review-graph` cuts Claude's token usage by up to 49x.
* Runs locally. No data leaves your machine.

* Problem: limited context window. On a repo with 27,000 files, the agent can't read everything.<br/>
  - The code agents fills the gaps with (often wrong) assumptions triggering 
    shallow reviews, missed side effects, hallucinated dependencies, ...

* code-review-graph uses Tree-sitter to parse the entire repo into a structural graph stored in SQLite. [[doc_has.keypoint]]
  - Every function, class, import, and dependency gets tracked.
  - When a file changes, it runs blast-radius analysis — tracing every affected file automatically.
    - Instead of scanning 27,000 files → it reads only 15.

The results:
- 6.8x fewer tokens on average code reviews.
- 49x fewer tokens on large monorepos.
- 2,900 files reindexed in under 2 seconds.
- Review quality score goes UP (8.8 vs 7.2)

- The bottleneck in AI-assisted coding isn't the model.
- It's not the code generation either.
- It's context. It's review. [[doc_has.keypoint]]

## RTK: Rust Token Kill

(Summary from post by Cerdá Enguix, -Kenmei.ai- on LinkedIN)

- rtk (Rust Token Killer) is a single binary rust client tool between your terminal and your AI agent.
  providing up 90% of token saving.

- It compares with other tools such as `lean-ctx` and  `Context+`. [[PM.price]]

- Each time the agent executes `git status` or `pytest` it receives hundreds of non-needed lines,
  comments, Boilerplate filling the context.
- RTK filters, groups, compress and deduplicates output from commands before reaching the LLM.
- <10millisecs of overhead.
- Fully transparent to agents.

  ```
  |                      Without RTK   With RTK
  | -------------------------------------------
  | git push              15 lines      1 line 
  |                      200 tokens    10 tokens
  | -------------------------------------------
  | cargo test           200 lines     20 lines
  | -------------------------------------------
  | 30 min.session  ~118000 tokens    ~23900
  ```

## GSD: LLM reasoning-scalability (reset token context)

- Every prompt you send makes Claude Code worse.
- Each message fills the context window. The more you type, the dumber it gets. (context rot)
- <https://github.com/gsd-build/get-shit-done/> GSD fixes it.
- One npm install. Works across Gemini CLI, Codex, and Copilot too.
- How it works: [[qa.best_patterns]]
  - Every task spawns fresh sub-agents with clean 200k token context.
  - Main session stays at 30-40% capacity while parallel agents handle execution.
  - Nothing accumulates. Nothing rots.

- workflow runs in four steps:
  1. Describe your idea fully
  2. System researches and plans automatically
  3. Parallel agents execute in waves
  4. Each task gets an atomic git commit

## Claude Best patterns

(Summary from Muhammad Haseeb post @ LinkedIn)

  Most developers use Claude Code in a loop:

  ```
  Prompt → Output → Repeat 
  ```

-  Then they wonder why the results feel inconsistent. 
- The real improvement does not come from writing better prompts every time,
  It comes from designing a project structure that gives Claude context, rules,
  and repeatable workflows.

  A practical setup looks like this:
  ```
  | <project root>/
  | +- CLAUDE.md     <·· the brain: project context, instructions, constraints 
  | +- skills/       <·· reusable workflows for review, refactor, release 
  | +- hooks/        <·· guardrails and automated checks
  | +- docs/         <·· architecture decisions so the "why" is documented
  | +- src/          <·· actual product code

  ```
 
- best AI workflows are not built on prompts alone.
- They are built on systems.
- That's when Claude Code stops acting like a tool and starts contributing like 
  part of an engineering environment.

## Claude Code Routines (2026-05)

* <https://www.infoq.com/news/2026/05/anthropic-routines-claude/?utm_content=051920>

- Routines allows developers to configure automated coding workflows that
  run on schedules, through API calls, or in response to external events.
- Routines runs on Claude Code’s cloud infrastructure (no need for local
  cron jobs, servers, or automation pipelines).  [[architecture.batch]]
- A routine consists of:
  - a prompt.
  - repository access.
  - connected tools or services.

- Example ussage: 
  - automated issue triage. 
  - documentation updates.
  - scanning documentation drift.
  - generating pull requests.
  - deployment verification.
  - alert analysis.
  - cross-language / cross-SDK synchronization.
  - ...

- API-triggered routines expose endpoints and authentication tokens,
  enabling external systems such as deployment pipelines, monitoring platforms,
  or internal tooling to trigger Claude Code sessions through HTTP requests.

- Webhook-based routines are added for GitHub events.

- Routines compare to Schedules in Claude Cowork, which run on the local machine.

- The release reflects a broader trend toward asynchronous AI coding agents that 
  operate continuously in cloud environments instead of interactive local 
  sessions. [[architecture.async]]

## 27 Claud Code Concepts Explained

1. agent == chatbot + (ReAct) actions (files, websites).
1. Permissions: Safety gate. Pre-approve safe moves (read/test) in 'settings.json'
1. Tools inlcude bash, local FS + MCP Server "plugins"
1. session == chat  == "short term memory"
   ```
   $ claude --verbose # <·· start with fress new short-term memory
   $ claude --resume  # <·· Inject previous session 
   $ claude -p .....  # <·· Batch mode (non-interactive)
   /help
   /compact           # <·· Compact Context (summarize and clear noise).
   /clear             # <·· clear short-term memory
   /model             # <·· Choose model. WARN: the most expensive is choosen by default.
                            Haiku (fast/cheap), Sonnet (all-rounder), Opus(genius)
                            --model (flag)
   /stats             # <·· track token usage+cost`
   settings.json      # <·· 
   /init
   /re                # <·· rewind mistakes [[agents.claude.troubleshooting]]
   ```
1. Skills: Expert playbooks (eg, Copywriting Skill, null killer skip, ...).
1. Hooks: Automatics 'if-this-then-that' rules (eg auto-formatting).
1. Sub-Agents: Delegation. Specialist run in clean windows.
1. Sub-Agent Teams: Collaboration. Multiple sub-agents with a shared goal.
1. Extended Thinking: Dedicated reasoning budget before action.
