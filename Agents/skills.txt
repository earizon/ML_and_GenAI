# Skills.md spec

* Originally developed by Anthropic but portable to any other agent framework.
* Main use-case: Define repeatable workflows (weekly reports, auto-triggered
  QA procedures, ...)
* Skills can be generic or specific to a project or organization (ex: 
  standarized procedures,...). Example skills.
  * `changelog/SKILLS.md`      : "here are the standarized formats we use, 
     here's the important information topics we cover, this is the layout 
     or structure expected, use a formal impersonal tone, here are the
     topics that must not be included, ..." 
  * `qa/SKILLS.md`             : standarized or repeatable Quality Assurance skills, linting,
  * `market-analysis/SKILLS.md`: Weekly analysis of market.
  * `campaign-briefs/SKILLS.md`: Common procedures to prepare a campaign.
* TIP: skills directories use lowercase with hyphens -vs spaces- by convention ("Kebab case").
* Most agents comes with pre-installed or "batteries-inside" skills.
* Skills can be invoked in several ways.
  1. Mention them explicetely in your prompt.
  2. According to the skills open standard, agents use the skill description to 
     decide when to call it in a process called progressive disclosure ... <br/>
     **WARN**: But their judgment isn't always perfect, especially as the context
     window gets larger. If you know you want a skill used, name tell it explicetely.
  3. An skill can also be mentioned into another skill.
* Sometimes skills can completely replace MCP servers, for example for small databases
  we can be stored as `.../my_skill02/references/database.csv`  (versioned in git, ...)
  removing the need for an MCP server querying an external database.
  [[doc_has.comparative,agents.MCP]]


* TIP: Many agents are moving from custom "/custom_command" to skills.

- For each "discovered" skill, agents will add them to an internal catalog, 
  a "simple" database that registers the metadata.name, metadata.description and,
  optionally, the path location of the SKILL.md file. This catalog is then 
  "injected" into the PROMPT, adding around 50-100 tokens per skill.<br/> 
  Modern LLMs with "big" contexts will have no problems with Agent's catalog 
  including dozens of skills.  [[agents.skills.catalog,doc_has.keypoint]]
  - Most probably it will end up being part of a system prompt, preceded by brief instructions
    on how to use skills. (simplest approach, it works with any model that has access to a 
    file-reading tool). The prompt could be something like:

  ```
  |  Next skills provide guided instructions for requested tasks.
  |
  |  When task matches a skill's description, use your file-read tool
  | to load the SKILL.md at the listed location before proceeding.
  |
  |  When a skill references relative paths, resolve them against the skill's
  |  directory (the parent of SKILL.md) and use absolute paths in tool calls.
  ```

- Alternatively, the catalog can be added into a dedicated skill-activation tool as part of 
  its description, keeping the system prompt clean. The tool description could be 
  something like:
  ```
  | The following skills provide specialized instructions for specific tasks.
  | When a task matches a skill's description, call the activate_skill tool
  | with the skill's name to load its full instructions.
  ```

Agents must be able to exclude skills from catalog due to different reasons including:

- System wide or explicit denial from user to use the skill.
- compatibility does not match running context (model, external tools, ...)
- ...

In such scenario, the skill must be omited from the (system prompt) catalog.

## Skill activation

(Step 4 @ <https://agentskills.io/client-implementation/adding-skills-support>)

Either the (LLM) model or the user can decide to use the skill (tier 2 of progressive disclosure). 


### Model-driven skill activation

Two implementation patterns:
  ```
  | title: File-read tool activation (standar agent tool)
  |
  | paritipant "File-read Tool" as FRT
  | Agent -> LLM  : prompt01
  | LLM   -> Agent: (some result indicating the Agent to use SKILL.md)
  | Agent -> FRT  : Read SKILL.md
  | FRT   -> Agent: SKILL.md
  | Agent -> Agent: skill = Replace rerences in SKILL.md
  | Agent -> Agent: prompt02 = prompt02 + skill
  | Agent -> Agent: prompt02 = prompt02 + ... (update with KV memory, ...)
  | Agent -> LLM  : prompt02
  | ...
  ```

  ```
  | title: Dedicated tool activation: 
  |
  | note over LLM,Agent: 
  |   function activation_tool takes an skill's name and returnes the effective content.
  |   required when the model can’t read files directly, and optional (but useful) even when it can. Advantages over raw file reads:
  | end note
  | ... -> Agent : Register activation_tool(skill_name) -> SKILL.md effective content
  ```

TIP: When the skill is used through a dedicated activation tool (vs enforced by user in prompt or discovered by the LLM)
     consider wrapping skill content in identifying tags. For example:
  ```
  | <skill_content name="pdf-processing">
  | # PDF Processing
  | 
  | ## When to use this skill
  | Use this skill when the user needs to work with PDF files...
  | 
  | [rest of SKILL.md body]
  | 
  | Skill directory: /home/user/.agents/skills/pdf-processing
  | Relative paths in this skill are relative to the skill directory.
  | 
  | <skill_resources>
  |   <file>scripts/extract.py</file>
  |   <file>scripts/merge.py</file>
  |   <file>references/pdf-spec-summary.md</file>
  | </skill_resources>
  | </skill_content>
  ```
Tags benefits:
- They allow the model to clearly distinguish skill instructions (vs conversation content)
- "Surface" Bundled resources to the model without being eagerly loaded.
- enhance context compaction (before sending to the model).
